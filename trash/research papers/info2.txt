# 2 — What you must analyze with **R** (and why)

R = statistical analysis, hypothesis testing, deep analytics, validating the model, and producing publication-quality plots.

**Main R analyses (produce notebooks / scripts):**

1. **Descriptive statistics & data health**

   * Distributions: temp, humidity, door durations, time_outside_range.
   * Missing data and sensor noise profile.
   * Sensor calibration offsets.

2. **Feature validation & engineering**

   * Check correlation matrix.
   * PCA / variance explained (optional) to validate features.
   * Test whether derived features (TSI, CRI, door severity) actually separate classes.

3. **Model validation**

   * Confusion matrix & class-wise precision/recall.
   * ROC/AUC for risk score binary thresholds.
   * Calibration plots (predicted probability vs observed frequency).
   * Lead-time analysis: compute minutes that model flags risk before threshold alarm — show distribution and mean lead-time.

4. **Anomaly detection / change-point analysis**

   * Use change-point detection to find sudden shifts in variance or mean (R packages: `changepoint`, `anomalize`).
   * Useful to detect sudden refrigeration failure or burst heating.

5. **Shelf-life / survival analysis**

   * Treat shelf-life consumption as a survival problem: time-to-excursion or time-to-spoilage. Use Kaplan–Meier curves for scenarios / routes.

6. **Hypothesis testing & significance**

   * Test whether door_open_count significantly increases risk (t-test / Wilcoxon / regression).
   * Compare routes: is Route A riskier than Route B (paired tests).

7. **Explainability & feature importance**

   * Coefficient significance (logistic regression p-values).
   * Partial dependence plots or simple marginal effects to explain how features change risk.

8. **Aggregated analytics for operations**

   * Daily/weekly excursion rates, avg recovery time, % trips with HIGH_RISK.
   * Predictive maintenance signals: trends in CRI over time per device.

**Deliverables from R:**

* PDF/plots for report (lead-time histograms, ROC, calibration, feature correlations)
* CSV reports for Power BI (aggregated / cleaned analytics)
* Scripts that produce the dataset used for retraining

# 5 — Streaming, ingestion & do you need Kafka?

Short answer: **No, not for student/demo scale.** Use MQTT + a simple ingestion backend. Use Kafka only if you need enterprise-scale guaranteed processing.

**Recommended ingestion stack (student/demo):**

* ESP32 → **MQTT broker** (Mosquitto) or HTTP POST to FastAPI
* FastAPI (or Flask) subscribes to MQTT or receives HTTP
* FastAPI writes raw data to **TimescaleDB** (Postgres + time-series) and writes windowed features (or receives them from ESP32) to `window_features` table
* Optional: push simplified events to Power BI via REST or allow Power BI pull

**When use Kafka / Spark?**

* Use **Kafka** if:

  * you simulate hundreds/thousands of devices producing thousands of events/sec
  * you want guaranteed ordering & replay and multiple consumers (analytics, alerts, ML)
* Use **Spark Structured Streaming** if:

  * you must process large-scale streams and do distributed feature computation, ML retrain at cluster scale

For capstone: mention Kafka+Spark as a possible production architecture and include a short diagram; implement MQTT+FastAPI.

# 6 — Storage & distributed DB options (recommended & production)

**Two levels: student/demo vs production**

### Student / Demo (recommended)

* **TimescaleDB (Postgres extension)** — time-series optimized, SQL-based, easy to query by Power BI via ODBC/gateway. Good for windowing, downsampling, retention.
* **MinIO** (S3-compatible) for storing long CSVs, model artifacts, and backups.
* Why: easy, free, integrates with R and Power BI.

### Production / Distributed (if you mention it)

* **Apache Spark** on a cluster for distributed batch/stream processing + MLlib or PySpark training.
* Store raw events as **Parquet** on HDFS or cloud object store (MinIO / S3) + maintain a Delta Lake for ACID + time travel.
* For real-time analytics use **Kafka** + **Spark Structured Streaming** and persist into ClickHouse / Cassandra / Druid for fast OLAP queries.
* For ML feature store & serving: **Feast** or a custom store using Redis for online features.

**Simple mapping:**

* Raw high-frequency → TimescaleDB or Parquet
* Aggregated feature windows → TimescaleDB (window_features table)
* ML training data -> Parquet/CSV in object storage -> Spark/Host for retraining

# 7 — How preprocessing & ML inference fit in the architecture

There are two valid approaches: **Edge-first** (we did) or **Backend preprocessing**.

We recommend Edge-first (ESP32 creates windows and features and runs inference). Backend receives features and prediction.

**Data flow:**

1. ESP32 collects per-second data.
2. Every 5 minutes, ESP32 computes feature vector and runs model.
3. ESP32 sends JSON {features, risk, window_start} to backend via MQTT/HTTP.
4. FastAPI stores raw events and window_features into DB.
5. R analytics jobs read from DB and produce validation reports.
6. Power BI reads aggregated views / SQL tables and renders dashboards.

This approach minimizes backend compute and preserves offline capability.

# 8 — KPIs & metrics to compute & present (exact formulas / definitions)

Include these on Power BI & in R analysis:

* **Lead time advantage** = mean( time_model_alert - time_threshold_alarm ) when model alerts before threshold.
* **Detection Recall (High Risk)** = TP_high / (TP_high + FN_high)
* **False Positive Rate** = FP / (FP + TN)
* **Shelf-life consumption proxy** = integral of risk_score over time normalized to %/trip (you can define a mapping risk→%/hour).
* **Time to recovery (CRI)** = time in minutes from excursion end to return within safe range.
* **Thermal Stability Index (TSI)** = std(temp) over sliding window or RMS of fluctuations.
* **Door abuse index** = weighted sum (door_count * avg_duration * max_duration)
* **Route risk score** = average risk_prob per km or per hour for the route
* **Device health score** = rolling average of CRI and peak temps

Implement these as SQL views in TimescaleDB for Power BI direct queries.
